# ETL-pruned

[中文版本 (Chinese)](./README.zh-CN.md)

`ETL-pruned` is a mirror repository used to publish a **submodule-friendly release branch** for [ETLCPP/etl](https://github.com/ETLCPP/etl).

## Why this repository exists

Some business repositories want to import ETL as a Git submodule, but only need the pruned release content.

This repository keeps a `release` branch that contains only:

- `include/etl/` contents (flattened to repository root)
- `LICENSE`
- upstream `README.md`
- `version.txt`

This makes the branch lightweight and stable for direct submodule integration.

## Usage

### Initialization

Add the `release` branch as submodule in your business repository:

```bash
git submodule add -b release https://github.com/inaku-Gyan/ETL-pruned.git third_party/etl
git submodule update --init --recursive
```

### Update to the latest published release

```bash
# Update the Submodule
cd third_party/etl
git fetch
git checkout release       # Checkout the latest version.

# Commit in your main repo
cd ../..
git add .
git commit -m "chore: update ETL submodule"
```

## How releases are produced

GitHub Actions workflow: `.github/workflows/release.yml`

When manually triggered (`workflow_dispatch`) with:

- `tag`: ETL tag in `X.Y.Z` format (required)
- `commit_message`: optional extra message

the workflow will:

1. Validate the tag format (`X.Y.Z`).
2. Checkout upstream `ETLCPP/etl` at that tag with sparse checkout.
3. Checkout this repository’s `release` branch.
4. Clean old files from `release`.
5. Copy pruned release files into `release`.
6. Commit and push to `release`.

## Notes

- Do not develop features directly on `release`; it is generated by workflow.
- If you need a new ETL version, trigger the release workflow with a new upstream tag.
